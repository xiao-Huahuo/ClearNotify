## 第一章 需求分析

### 1.1 项目背景与问题提出

在日常学习、工作和生活中，政策通知、办事指南、申报公告、社区提醒、校园事务说明等信息经常以网页公告、PDF 附件、扫描件、拍照图片、聊天截图、表格附件等多种形式出现。此类信息往往包含大量条件限制、材料要求、时间节点、适用对象和办理路径，但其表达方式并不统一，很多内容仍然保留传统公文、通知或附件风格。对于普通用户而言，真正困难的并不是“是否看到了这份材料”，而是“是否能快速看懂、确认是否适用、明确下一步要做什么”。

现有的信息获取方式主要集中在公告浏览、关键词检索和零散咨询三个层面。一方面，信息来源分散在多个网站、公众号、群聊消息和附件文件中，用户需要自行判断哪个入口有效、哪份材料最新、哪些内容与自身相关；另一方面，即使找到了原始文件，仍然需要手工从长文本中提取材料、步骤、截止时间、办理入口和注意事项，理解成本和执行成本较高。对于需要准备材料、关注时效、反复核对条件的事项，用户很容易在信息阅读阶段就产生遗漏或偏差。

随着大模型和智能问答工具的发展，用户开始尝试将政策通知交给通用 AI 进行概括和解释，但政策、通知和办事说明属于高准确性场景，仅有自然语言总结并不能完全满足实际需要。真实使用中，用户更关心的是结果是否有依据、是否能继续追问、是否能结合原文核对、是否能沉淀为后续待办事项，以及是否可以在后续使用中复用同一份解析成果。因此，本作品希望构建的并不是单一的“问答工具”，而是一套围绕上传、解析、追问、执行、反馈和沉淀形成闭环的智能化服务平台。

云枢观策正是在这一背景下提出。作品聚焦公共信息理解与办事辅助场景，以多模态材料接入、自由格式图谱解析、知识图谱构建、RAG 检索增强、可解释智能体和结果沉淀机制为核心，面向真实使用过程中的信息获取、内容理解、执行跟进和反馈收集等环节，提供一套更完整、更稳定、更易落地的产品方案。

### 1.2 典型场景中的现实痛点

#### 1.2.1 信息来源分散，入口判断成本高

用户获取公共服务信息时，通常需要在多个来源之间切换，例如政策网站、地方公告页、学校通知页、群聊转发、截图消息、扫描件附件和压缩包材料。来源越多，用户越难判断哪一份最完整、哪一份仍然有效、哪一份与自己关系最密切。对于不熟悉政策检索习惯的用户，找到材料本身就已经耗费大量时间。

#### 1.2.2 长文本与附件材料理解成本高

很多政策性材料为了保持原始表达的完整性，会使用大量规范化、说明性、条件性语句。用户在阅读时最关注的是适用对象、办理时间、提交地点、所需材料、注意事项与可能风险，但这些信息往往分散在多段文字中，甚至埋在附件、附表或扫描图片里。单纯阅读原文并不能直接形成可执行的步骤清单。

#### 1.2.3 非结构化材料比例高

在实际使用中，信息并不总是规范的网页文字。用户经常接触到拍照版通知、扫描版 PDF、带有印章的图片材料、聊天截图、表格截图和图文混排附件。此类材料即便能够被 OCR 提取文字，也常常存在结构混乱、顺序失真和字段难定位的问题。传统“复制文字再手工整理”的方式效率较低。

#### 1.2.4 从“理解”到“执行”之间缺少衔接

即便用户已经看懂了一份通知，也不等于能够顺利办理。许多事项需要多步处理，包括准备不同材料、区分优先级、记录截止时间、跟进进度和再次核对原文。单次解析结果若无法沉淀为可复用内容，用户仍可能在之后重复阅读、重复提取和重复确认。

#### 1.2.5 用户反馈与内容优化缺乏闭环

用户在实际阅读和办理过程中往往会发现歧义点、遗漏点和高频困惑，但这些反馈很少能结构化沉淀下来。发布主体难以及时看到用户在使用中的问题，后续浏览者也无法提前获知哪些环节最容易误解、哪些要求最容易出错，导致信息服务停留在单向发布而不是持续优化。

### 1.3 目标用户与典型使用场景

本作品面向的不是单一用户群体，而是围绕公共服务信息流转全过程构建三类主要角色和一类管理角色。

#### 1.3.1 普通用户

普通用户是作品最核心的使用对象，包括居民、学生、职场人员及需要日常办理事务的人群。该类用户的主要需求包括：

- 快速读懂通知重点，而不是逐段翻阅全文
- 判断自己是否适用某项规则或办理事项
- 从复杂文本中提取所需材料、时间节点和办理步骤
- 在阅读结果基础上继续追问细节
- 将结果保存、收藏或转化为待办事项

典型场景包括：查看社区通知、阅读学校事务说明、处理拍照上传的公告、快速识别截图中的办事要求、理解政策性文件中的材料清单与限制条件等。

#### 1.3.2 企业与创业主体

企业主体特别关注申报条件、申报时限、所需材料、入口路径和后续跟进。相比普通用户，企业对信息准确性、时效性和执行链路的要求更高。其主要需求包括：

- 从长篇政策文本中快速提取可执行信息
- 对申报、申请、审核和补充材料流程形成清晰理解
- 保留解析与追问记录，便于多人协作和后续复用
- 在多份政策材料中识别更适合自身的内容

#### 1.3.3 认证发布主体

认证主体是平台中的内容发布与跟踪角色。其主要需求包括：

- 通过统一入口上传并发布政策文件或通知材料
- 对发布内容进行分类整理和公开展示
- 追踪内容被浏览、点赞、评价和纠错的情况
- 通过用户反馈发现说明不清、要求不明确或高频疑问点

#### 1.3.4 管理员与运营者

管理员负责平台整体秩序、内容审核和数据观察。其主要需求包括：

- 查看用户角色结构、总体活跃度和解析规模
- 审核发布内容并控制公开展示节奏
- 查看反馈类型分布、热点内容流转和统计分析结果
- 通过统一后台掌握系统运行与内容沉淀状况

### 1.5 本作品的需求定位

云枢观策的需求定位可以概括为：围绕复杂公共信息的理解、执行和反馈过程，构建一套支持多模态输入、自由格式图谱解析、证据增强、持续对话和结果沉淀的智能化服务平台。

该定位主要体现在以下几点：

- 作品处理的不是单一纯文本，而是现实使用中常见的 PDF、Word、Excel、图片、截图、URL 等多种材料
- 作品目标不是停留在“把文章总结短一点”，而是将长文本转化为更适合阅读、核对和执行的自由格式图谱结果
- 作品不仅提供一次性结果，还提供图谱展示、追问、待办、收藏、历史恢复和公开反馈能力
- 作品同时面向普通用户、认证主体和管理员，具备多角色业务逻辑
- 作品既适合展示，也具备可持续使用的系统结构

### 1.6 本作品的核心优势与应用价值

#### 1.6.1 多模态接入能力

作品能够接入真实使用环境下的多种输入形式，包括网页文本、附件文档、截图、扫描件和图片，减少用户将材料手工整理为标准文本的前置成本。

#### 1.6.2 可解释的智能体交互能力

作品中的智能体不仅提供对话式追问能力，还通过实时推理轨迹和证据展示降低结果的不可解释性，使用户在使用时更容易建立信任和判断依据。

#### 1.6.3 从“理解”到“执行”的结果承接能力

作品将解析结果与收藏、历史记录、待办事项、发布反馈等模块打通，能够把一次性阅读结果沉淀为后续持续可用的内容资产。

#### 1.6.4 平台级产品结构

作品并非单页面效果演示，而是形成了展示入口、业务入口、发布中心、反馈大厅和管理员后台共同组成的平台化结构，更容易在校园、社区、园区、服务窗口等场景中延展使用。

## 第二章 概要设计

### 2.1 设计目标与总体思路

云枢观策的总体设计目标，是围绕复杂公共信息的获取、理解、追问、执行与反馈过程，构建一套可展示、可运行、可持续扩展的智能化服务平台。设计中遵循“看得懂、找得到、办得成、能反馈、可沉淀”的总体思路。

在这一思路下，系统需要同时解决以下问题：

- 如何统一接入不同来源和格式的材料
- 如何把非结构化内容转为更适合执行的自由格式图谱结果
- 如何通过智能体持续承接用户追问，而不是只给一次性结果
- 如何保留原文、解析结果、证据和轨迹之间的对应关系
- 如何把一次解析扩展为收藏、待办、历史和反馈等后续行为
- 如何同时满足作品展示、业务使用和管理分析三种需求

因此，作品在产品层采用双入口结构，在技术层采用前后端分离与智能中枢协同，在数据层采用业务数据、统计数据与知识增强数据并行管理的方式，形成一套完整架构。

### 2.2 系统总体架构

系统总体上可以划分为五层：

1. 表现层：由 Vue3 前端构成，负责展示首页、业务页面、智能体页面、图谱结果页面、管理后台和数据可视化页面。
2. 接口与业务层：由 FastAPI 后端提供 REST API 与 WebSocket 服务，承接上传、鉴权、政策文档、反馈、历史记录、统计分析等业务逻辑。
3. 智能中枢层：由文档解析、OCR、RAG、Agent 插件、实时轨迹编排等模块组成，负责非结构化内容理解和智能体交互。
4. 数据与知识层：包括 SQLite 业务数据库、ChromaDB 向量库、初始化知识资源和统计分析结果。
5. 运维与部署层：包括 Docker、Docker Compose、Nginx、Redis、日志目录和邮件输出目录。

### 2.3 双入口产品结构设计

为了兼顾展示效果和真实使用，作品没有将展示与业务拆成两个独立项目，而是采用双入口并行的设计。

#### 2.3.1 展示入口

根路由默认跳转至 `/showcase`。展示入口承担以下职责：

- 对外说明作品定位、核心亮点和整体能力
- 承接答辩、展示和路演过程中的场景说明
- 通过展示版发现页和展示大屏强化作品的整体感与完成度
- 引导用户从“看项目”自然过渡到“用系统”

#### 2.3.2 业务入口

当用户进入系统后，默认落到 `/agent` 智能体工作区，并可进一步进入主页解析区、政策广场、待办页、历史页、收藏页、反馈大厅、发布中心和管理后台等页面。

这一结构的优点在于：

- 展示层适合评审快速理解系统全貌
- 业务层适合真实演示上传、解析、追问和沉淀链路
- 两者位于同一产品体系中，避免展示逻辑和业务逻辑割裂

### 2.4 功能模块划分

依据当前产品结构，系统可以划分为六个核心模块。

#### 2.4.1 多模态解析工作台

负责接收用户上传的 PDF、Word、Excel、TXT、图片、截图和 URL 信息，并在解析过程中同步反馈进度、状态和结果展示。

#### 2.4.2 智能体系统

负责承接持续追问、RAG 证据增强、工具调用、实时轨迹展示、会话管理和消息持久化。

#### 2.4.3 政策广场与发现体系

负责承接公开展示内容，包括审核通过的政策文档、热点资讯、推荐阅读和内容详情页。

#### 2.4.4 结果沉淀与办事承接模块

包括收藏、历史记录、待办事项和个人中心，使用户能够将解析结果转化为后续可持续使用的内容。

#### 2.4.5 认证主体发布与反馈闭环模块

包括政策发布中心、发布数据追踪和反馈大厅，用于支撑认证主体上传发布、用户评价纠错和使用反馈观察。

#### 2.4.6 管理后台与数据分析模块

包括用户管理、权限管理、数据汇总、全局分析、日志查看、RAG 状态和展示大屏数据等能力。

### 2.5 用户角色与业务流程

系统围绕普通用户、认证主体和管理员三类角色设计不同链路。

#### 2.5.1 普通用户链路

普通用户的典型使用过程如下：

上传材料或输入文本 -> 系统解析并生成自由格式图谱结果 -> 用户查看图谱、重点信息和原文 -> 在智能体页继续追问 -> 将结果收藏或转为待办 -> 在历史页中恢复和复用结果 -> 对公开内容提交意见或评价。

#### 2.5.2 认证主体链路

认证主体的典型使用过程如下：

上传政策文档 -> 填写标题、正文、分类和标签 -> 进入待审核状态 -> 审核通过后在政策广场公开展示 -> 用户浏览、点赞、评价和纠错 -> 认证主体在追踪页查看效果和反馈。

#### 2.5.3 管理员链路

管理员的典型使用过程如下：

管理用户与角色 -> 审核待发布文档 -> 查看全站统计分析 -> 查看意见分布和热点趋势 -> 监控日志与 RAG 状态 -> 借助数据大屏完成展示与汇总。

### 2.6 作品特色与创新设计概述

从总体设计角度看，本作品的创新不主要体现在单一模型或单一算法的使用上，而体现在围绕真实使用过程构建了一条完整且可运行的业务闭环，并在这一闭环中将自由格式图谱解析、可解释智能体、多模态材料处理、证据增强、结果沉淀和反馈治理整合为同一套系统。相较于仅提供查询、摘要或聊天功能的常见方案，云枢观策更强调“材料进入系统之后，如何持续服务后续理解、执行、复用和反馈”的全过程能力。

#### 2.6.1 创新点一：从单次解析升级为完整业务闭环

本作品最核心的创新点，是把公共信息处理从“看一份通知”扩展为“上传材料、完成解析、继续追问、形成结果、推进执行、接收反馈、持续沉淀”的完整闭环。系统不是在页面上孤立提供一个上传按钮或对话框，而是将多个后续动作自然连接起来：

- 材料进入系统后，不只是得到一次性回答，而是形成可继续追问的结果上下文  
- 解析结果不只停留在当前页面，还可以进入收藏、历史记录和待办链路  
- 用户对公开内容的评价、纠错和留言能够继续沉淀到反馈链路中  
- 认证主体可以从发布、展示、追踪到观察反馈，形成完整内容运营路径  
- 管理员能够基于用户、内容和反馈数据进行审核和全局分析  

这一设计使作品从“智能工具”提升为“平台型系统”。从系统定位上看，作品所解决的并非单点阅读问题，而是复杂公共信息在真实使用过程中的前后承接问题。正因为构建了这一闭环，系统才具备持续使用价值，而非停留在一次展示后即结束的功能集合。

#### 2.6.2 创新点二：以自由格式图谱解析取代固定模板抽取

现实中的政策与通知材料结构差异极大，若采用固定字段模板，往往只能抽取有限信息，且容易在结构不规范、内容混杂或条目密集的场景下出现表达失真。为解决这一问题，本作品没有把解析目标限定为若干固定字段，而是采用自由格式图谱解析思路：

- 先生成 `dynamic_payload` 形式的自由格式内容载荷  
- 再从载荷反推 `nodes / links`，形成知识图谱结构  
- 最终实现“图谱浏览 + 节点定位 + 原文联动 + 后续继续处理”的统一结果体系  

这种设计的创新意义在于，它不要求所有材料都服从同一模板，而是允许不同类型的文本以更接近原始语义结构的方式进入图谱结果。相比传统摘要卡片，自由格式图谱解析更适合表达流程、条件、对象、限制和步骤之间的关系，也更适合在复杂材料场景下保留信息层次。

#### 2.6.3 创新点三：将可解释智能体嵌入结果闭环

本作品中的智能体并非只负责“回答问题”，而是承担理解、追问、证据调用、轨迹展示和结果承接的综合角色。智能体能力并不是孤立悬浮在页面上，而是与图谱结果、历史会话、证据检索和用户后续动作深度结合。

其创新性主要体现在：

- 智能体可以围绕已上传材料继续追问，而不是脱离上下文重新开始  
- 交互过程中不仅返回正文，还展示实时推理轨迹和工具调用步骤  
- 结果与证据、轨迹、会话记录保持一致，使用户能够在不同层次上核对内容  
- 智能体输出可以继续进入收藏、历史和待办，而不是停留在聊天气泡中  

这种做法让智能体从“单轮回答接口”升级为“闭环中的持续交互中枢”，既增强了系统的解释性，也增强了系统在真实使用中的连续性。

#### 2.6.4 创新点四：多模态材料统一进入同一理解链路

许多实际使用场景下，信息来源并不是标准文本，而是图片、扫描件、截图、表格附件和图文混排文档。很多系统虽然支持上传多种格式，但不同格式进入的是不同的功能分支，最终结果难以统一复用。本作品的创新点在于，不同材料虽然采用不同提取路径，但最终会统一进入同一条理解链路：

多模态输入 -> 文本抽取 / OCR -> 自由格式图谱解析 -> 智能体追问 -> 结果沉淀

这种统一处理方式带来的直接效果包括：

- 用户不需要判断“某种材料该去哪个子系统处理”  
- 图片、截图、文档和文本可以进入统一结果体系  
- 后续收藏、历史、待办、反馈等能力不再依赖材料来源类型  
- 系统更接近真实使用环境，而不是只适合标准样例输入  

由此可见，作品的创新并不在于单纯支持多种格式，而在于让多种格式最终进入同一个产品逻辑和结果闭环。

#### 2.6.5 创新点五：RAG 从后台增强能力变成前台可感知能力

很多系统使用 RAG 时只把它作为后台增强策略，对用户而言只能感受到“回答可能更准”，却无法知道系统到底检索到了什么。本作品则将 RAG 的作用从内部逻辑前移到了交互层：

- 系统会对知识片段执行语义检索和阈值筛选  
- 证据片段和相似度分数能够在界面中被显式展示  
- 管理端可以进一步观察命中率、平均分数和空检率等指标  

这意味着“有依据”不再只是系统开发层的内部承诺，而成为用户可感知、管理员可观察的实际能力。对需要较高可信度的理解场景来说，这种设计比单纯依赖大模型生成更具说服力。

#### 2.6.6 创新点六：展示入口与业务入口的一体化产品结构

本作品并未将展示页和业务系统拆分为两个独立站点，而是通过 `/showcase` 展示首页、展示版发现页、展示大屏与主业务系统共同构成一套连续访问链路。这样做的优势在于：

- 展示层适合说明作品定位、亮点和完整性  
- 业务层适合直接演示上传、解析、追问和沉淀过程  
- 两者同域协同，减少了“比赛展示版本”和“实际使用版本”脱节的问题  

对于参赛作品而言，这种结构既有利于答辩和演示，也能体现作品并非概念展示，而是具备实际产品形态。

#### 2.6.7 创新点七：技术创新服务于闭环，而非孤立堆叠

在技术层面，本作品确实使用了自由格式图谱解析、RAG 检索增强、WebSocket 实时轨迹、多模态 OCR、统计建模、Docker 部署等多项能力，但这些能力的价值不在于单独列出“用了哪些技术”，而在于它们共同服务于前述闭环。具体来看：

- 自由格式图谱解析解决复杂材料难以稳定表达的问题  
- 智能体与实时轨迹解决结果不可解释、不可持续追问的问题  
- 多模态解析与 OCR 解决现实输入来源复杂的问题  
- RAG 证据增强解决依据不足和结果可信度的问题  
- 收藏、待办、历史与反馈模块解决结果无法继续使用的问题  
- 展示层、发布层和管理层解决系统只能“自己用”而不能“共同使用、观察和管理”的问题  

因此，本作品的技术创新具有明显的系统集成特征。它不是围绕某个独立算法做封闭优化，而是围绕一条完整服务链路，把多个技术点组织成一套可运行、可展示、可扩展的平台方案。

## 第三章 详细设计

### 3.1 系统技术架构设计

#### 3.1.1 前端架构

前端采用 Vue 3 + Vite 组合，使用 Composition API 组织页面和逻辑，并以 Pinia 统一管理鉴权状态、用户信息、主题模式和界面偏好。Vue Router 负责页面路由及管理员页面的权限拦截。ECharts 用于统计分析图表与展示大屏可视化。

前端架构中，页面视图集中放置在 `web/src/views`，组件主要放置在 `web/src/components`，路由统一收敛在 `web/src/router/index.js`，接口地址统一通过 `web/src/router/api_routes.js` 管理。该组织方式便于在比赛展示和后续迭代中维持较清晰的前端边界。

#### 3.1.2 后端架构

后端采用 FastAPI 异步架构，整体可划分为接口层、业务服务层、智能处理层、数据模型层和基础设施层：

- 接口层：位于 `app/api/routes`，负责定义 REST 接口和 WebSocket 路由
- 业务服务层：位于 `app/services`，负责统计、解析、邮件、抓取、OCR、Agent 协同等功能
- 智能处理层：位于 `app/ai` 与 `app/agent_plugin`
- 数据模型层：位于 `app/models` 与 `app/schemas`
- 基础设施层：位于 `app/core`，负责数据库、日志、配置、安全等

#### 3.1.3 智能中枢协同方式

智能中枢由文档解析、OCR、RAG、Agent 插件和实时轨迹编排共同组成。普通解析任务可由文件上传接口触发，复杂追问和持续会话由 Agent 路由与 WebSocket 服务承接。RAG 作为知识增强机制贯穿智能体回答与自由格式图谱结果校准过程。

### 3.3 多模态解析工作台设计

- 将 OCR 提取与智能解析衔接起来，使图片类内容也能进入统一结果链路

### 3.4 智能体系统设计

智能体系统围绕两个接口形态构建：

- HTTP 接口 `/agent/run`：用于一次性执行智能体任务并返回自由格式图谱结果
- WebSocket 接口 `/agent/ws`：用于会话式流式交互和实时轨迹展示

同时，系统还实现了以下辅助接口：

- 会话列表获取
- 新建会话
- 删除会话
- 历史消息读取

在数据层，对话与消息分别由 `AgentConversation` 和 `AgentMessage` 两张表持久化。这样既能维持完整会话记录，也方便前端按会话粒度恢复历史状态。

### 3.5 知识图谱与结果呈现设计

系统的结果呈现不只是文本摘要，而是强调图谱化和自由格式回显。解析过程中，系统会尝试输出：

- `nodes`
- `links`
- `dynamic_payload`
- `visual_config`

其中，`dynamic_payload` 用于描述解析出的自由格式内容，`nodes` 和 `links` 则进一步将其映射为前端可展示的图谱结构。当前设计支持：

- 节点与原文映射
- 父子关系或步骤关系展示
- 图谱结果与原文回看联动
- 将自由格式图谱结果用于后续推荐、统计和办事沉淀

这种结果设计比传统卡片式摘要更适合处理结构差异大、逻辑层次多的材料。

### 3.6 文档解析与 OCR 处理链设计

系统在文档处理上采用“文本优先、OCR 兜底、多路径回退”的策略。

#### 3.6.1 文档解析路径

- PDF：优先使用 `pdfplumber` / PyMuPDF 提取文本块
- DOCX：使用 `python-docx`
- XLSX：使用 `openpyxl`
- XLS：使用 `xlrd`
- TXT：直接读取文本

#### 3.6.2 OCR 路径

图片、扫描件和截图进入 OCR 流程后，由 `ocr_service` 统一处理。当前实现以 Kimi 文件提取和视觉识别能力作为 OCR 主要链路。对于文件提取结果，系统还会执行文本标准化与结构清理，以便更平滑地进入后续解析阶段。

#### 3.6.3 回退与容错策略

在面对图文混排 PDF、多图文档、结构异常内容时，系统会采取文本块与图片块混合处理、提取结果规范化和失败回退等策略。这样的设计避免了单一路径失败后整条链路中断。

### 3.8 数据库设计

系统数据库采用 SQLite 持久化，利用 SQLModel 构建业务实体。数据库设计并非只为“存数据”服务，而是围绕用户身份、解析过程、图谱结果、发布反馈和统计分析五类业务需求展开。整体上遵循“核心业务表负责主链路，辅助分析表负责沉淀和观测”的设计思路，既保证普通使用场景下的响应效率，也方便后续扩展统计、搜索和推荐能力。

从功能结构上看，数据库主要承担四个作用：

- 保存用户、角色、设置和登录验证等基础信息  
- 持久化解析结果、会话历史、收藏、待办和自由格式图谱结果的关联信息  
- 维护政策文档发布、意见反馈、公开展示和审核状态  
- 为全站分析、RAG 指标、搜索索引和个性化推荐提供数据底座  

#### 3.8.1 用户与权限相关实体

`User` 表是系统身份体系的核心实体，记录用户名称、邮箱、手机号、登录手机号、角色、职业、头像地址、最近登录方式、邮箱验证状态、手机验证状态和密码登录开关等字段。系统通过 `role` 字段区分普通用户、认证主体和管理员三类身份，并在后端接口与前端路由中共同使用这一字段完成权限控制。

`Settings` 表与 `User` 形成一对一关系，用于存储默认受众、明暗主题、全局色系和系统通知偏好。该表的存在使系统界面偏好不再依赖纯前端本地缓存，而能够在用户登录后跨设备同步，增强产品化一致性。

这一部分的数据库设计特点在于：

- 将身份、验证和偏好分离，避免单表字段持续膨胀  
- 角色与业务能力的映射清晰，便于后续新增角色  
- 个性化设置持久化后，可以直接服务展示页和业务页的一致主题切换  

#### 3.8.2 会话与历史相关实体

`ChatMessage` 表用于保存解析型任务产生的结果数据，是系统业务主链中最关键的数据表之一。除原始文本外，该表还保存适用对象、办理事项、时间节点、办理入口、材料清单、办理流程、注意事项、风险提醒、原文映射和通知分析结果等内容，并通过 `source_chat_id` 支持结果之间的继承与回溯。尽管自由格式图谱结果在展示层并不依赖固定模板，但这些字段仍然承担了高频摘要信息和统计分析的基础作用。

`AgentConversation` 与 `AgentMessage` 两张表共同构成智能体会话存储结构。其中前者按会话粒度保存标题、创建时间和更新时间，后者按消息粒度保存用户、角色和正文内容。这样的设计使智能体页面既能按会话列表快速恢复，又能在消息级别持续追加内容。

`HistoryEvent` 表承担统一历史轨迹记录职责，用于保存解析恢复、跳转入口、操作摘要、可恢复状态、可见性和去重键等信息。相较于简单的“最近记录”列表，`HistoryEvent` 通过 `domain`、`event_type`、`subject_type`、`status` 等字段提供了更灵活的历史过滤与聚合能力。

这一层设计的重点在于：

- 将“解析结果”和“智能体对话”分别建模，避免两类数据互相污染  
- 历史事件采用统一事件表，而非让各业务模块各自保存最近记录，便于形成通用的历史视图  
- 为结果恢复、对话延续和搜索索引提供清晰的数据来源  

#### 3.8.3 内容与反馈相关实体

`PolicyDocument` 表是公开内容体系的核心，记录文档标题、正文、分类、标签、上传者、审核状态、文件路径、浏览量、点赞量、审核时间和驳回原因。该表支撑了政策发布、审核、推荐、公开展示和发布效果追踪等多个功能页面，是“认证主体发布链路”的核心数据载体。

`Opinion` 表记录用户围绕公开文档产生的反馈内容，包括评价、纠错和留言三种类型，并保存评分、点赞量和关联文档信息。该表的设计使反馈不再只是页面评论，而成为可聚合、可统计、可追踪的结构化数据。

`Favorite` 表将用户和具体解析结果关联起来，提供个人沉淀能力；`TodoItem` 表则在用户与待办事项之间建立关系，额外保留来源解析记录、截止时间、是否确认、是否完成等字段，用于把“看懂内容”进一步转化为“推进事项”。

这一部分设计体现了作品的闭环思路：

- 内容不是发布后即结束，而是进入阅读、点赞、反馈和追踪链路  
- 解析结果不是展示后即丢弃，而是能够转入收藏和待办  
- 用户行为会反向沉淀为可继续使用的数据资产  

#### 3.8.4 统计与检索相关实体

`StatsAnalysis` 表负责保存统计分析结果，包括材料词频、风险词频、复杂度分布、通知类型分布、总解析量、累计节省时间和时间分布数据等。该表既支持用户个人统计，也支持全站统计，能够为个人中心、管理后台和展示大屏提供统一数据来源。

`RagUsage` 表记录每次检索增强的查询内容、`top_k` 参数、命中条数、平均相似度和来源，用于生成命中率、平均得分和空检率等指标。其价值在于把 RAG 从后台增强逻辑转化为可分析、可观测的数据对象。

`SearchIndexItem` 表承担统一搜索索引职责，用于将不同来源的数据标准化为可检索对象，包括标题、摘要、正文、关键字、路由地址、外链地址、可见性和嵌入向量字段等。这种设计便于后续将全站检索、推荐和聚合入口统一到同一套索引体系之中。

`AgentMemory` 表记录智能体会话摘要，是长期记忆与跨轮上下文承接的基础数据结构。它与 `AgentConversation` 的区别在于：前者强调摘要化的会话记忆，后者强调原始会话结构和页面恢复。

#### 3.8.5 核心数据关系说明

从关系上看，系统的关键实体之间形成以下主链路：

- `User -> ChatMessage -> Favorite / TodoItem / HistoryEvent`：用于承接用户解析、结果沉淀与后续使用  
- `User -> AgentConversation -> AgentMessage`：用于承接智能体连续对话链路  
- `User -> PolicyDocument -> Opinion`：用于承接发布、公开展示和反馈治理链路  
- `User -> Settings / StatsAnalysis / RagUsage`：用于承接个性化配置与统计观测链路  

这种关系设计有两个明显优势：

- 核心链路稳定。用户、内容、反馈、统计之间关系明确，便于扩展页面和接口。  
- 辅助链路独立。搜索索引、RAG 指标、历史轨迹和智能体摘要等功能可以并行增强，而不必频繁修改主业务表。

### 3.9 关键算法与核心技术

#### 3.9.1 自由格式图谱解析与图谱生成算法

系统没有将文档强行压缩成固定字段模板，而是采用“严格约束输出 + 宽松纠错修复 + 可展示回退”的自由格式图谱解析算法。其核心目标不是只提取少量摘要字段，而是在尽可能保留材料语义层次的前提下，形成可展示、可联动、可继续处理的图谱载荷。

这一算法主要包括四个步骤：

1. 使用 `json_schema / json_object` 等方式约束大模型输出，优先得到可直接解析的 JSON 结构。  
2. 当返回内容夹带 Markdown、解释前缀或尾部截断时，执行首个 JSON 对象提取、冗余逗号修复与截断补全。  
3. 若轻量恢复失败，则进入 Repair-Only 修复阶段，只修正 JSON 结构，不主动改写语义内容。  
4. 将 `dynamic_payload` 反推为 `nodes / links`，补齐节点标识、边关系、焦点节点和原文映射，形成前端可展示图谱。

在模型仅给出少量抽象节点时，系统还会根据原文行块、章节编号、段落与句群补出一层原文结构图层，使大文档在图谱视图中不会只剩少量标题节点，而是保留更高密度的结构信息。

#### 3.9.2 语义检索与置信校准算法

系统的 RAG 机制并不是简单“查资料后拼接上下文”，而是由本地 embedding、向量检索、阈值过滤和结果置信度估计共同组成。检索阶段通过 `top_k + score_threshold` 双阈筛选候选片段，减少低质量上下文进入生成阶段；展示阶段则把证据片段及其相似度分数显式呈现给用户和管理端。

为了估计解析结果的可信度，系统采用如下置信度加权模型：

$$
C = \min \left(0.95,\ 0.35 + 0.2 \cdot \mathbb{I}(\text{parse\_mode}=\text{ai}) + 0.45 \cdot \min(\bar{s}, 1)\right)
$$

其中：

- $C$ 表示最终置信度  
- $\mathbb{I}(\text{parse\_mode}=\text{ai})$ 表示解析是否由 AI 主导完成  
- $\bar{s}$ 表示证据片段平均相似度分数  

该公式的意义在于同时考虑“模型是否完成了完整解析”和“知识证据是否真正命中”，从而避免仅凭模型表述流畅度判断结果质量。

#### 3.9.3 图谱侧本地检索与个性化推荐算法

在图谱工作区中，系统不仅展示节点关系，还围绕当前用户需求进行局部检索和价值排序。前端会对节点标题、节点出处摘要执行前缀命中、包含命中与 Dice bigram 相似度计算，形成轻量级本地混合检索器，帮助用户在图谱内快速定位关键信息。

此外，系统显式计算当前文档对用户的推荐分数：

$$
R = 100 \cdot \frac{w_p S_{profession} + w_r S_{role} + w_a S_{action} + w_u S_{urgency} + w_c S_{complexity}}{\sum w}
$$

其中：

- $S_{profession}$ 表示职业短语及扩展关键词与文档正文、节点标签、动态载荷文本的匹配强度  
- $S_{role}$ 表示用户角色与文档语义重心的匹配程度  
- $S_{action}$ 表示“申请、申报、材料、入口、资格、流程”等可执行信号密度  
- $S_{urgency}$ 表示截止时间、逾期、处罚、风险等时效约束信号密度  
- $S_{complexity}$ 表示由节点数、关系数、负向约束数量和正文长度共同估计出的价值节省强度  

当前实现中，职业匹配、角色匹配、可执行性、时效/风险、复杂度节省价值的权重分别取 `0.30`、`0.22`、`0.24`、`0.14` 和 `0.10`。若用户未填写职业信息，则职业项权重自动归零，其余权重重新归一化，避免空字段直接拉低推荐质量。

#### 3.9.4 实时轨迹调度与流式消费模型

WebSocket 交互中，系统将推理轨迹事件放入队列，优先推送思考步骤和工具事件，待轨迹结束后再推送正文结果。该策略解决了“正文输出过快导致最后几步轨迹被覆盖”的问题。

后端通过 `asyncio.Queue + asyncio.to_thread` 形成“主协程负责推送、工作线程负责推理”的双通道调度模型；前端则通过最小可读时长策略控制轨迹消费节奏，防止工具步骤快速闪过，增强页面的可读性和稳定性。

#### 3.9.5 复杂度评估与轻量评分模型

系统在首页、历史页、发现页和管理端引入复杂度评估机制，用于支持排序、标签展示和分析看板。其核心思路是把文档长度、图谱节点规模、关系密度和风险提示强度共同映射为更容易理解的难度指标。

综合难度因子定义如下：

$$
D = \alpha \cdot \ln(W_{original}) + \beta \cdot N_{entities} + \gamma \cdot \sigma_{logic}
$$

其中：

- $W_{original}$ 表示原文字符量  
- $N_{entities}$ 表示材料、步骤、对象等实体规模  
- $\sigma_{logic}$ 表示限制条件、否定关系和流程嵌套强度  

在当前实现中，这些量通过图谱边数、风险等级、材料项数量和结构标签等指标进行近似映射，从而支持“低 / 中 / 高”复杂度的前端展示与后台分析。

#### 3.9.6 效率增益与统计建模体系

为了量化系统对阅读理解过程的帮助，平台引入时间节省估算、协同效率指数和词频/检索表现统计等模型。时间节省估算模型如下：

$$
T_{saved} = \max \left( 0, \frac{W_{original}}{V_{human}} - T_{card} \right)
$$

其中：

- $W_{original}$ 表示原文字符量  
- $V_{human}$ 表示常态阅读理解速度  
- $T_{card}$ 表示用户阅读自由格式图谱结果所需平均时间  

该模型用于估算一次解析相对于直接阅读原文所节省的理解时间。为了避免极端长文本导致估值失真，系统在实际实现中还会加入上下界裁剪。

在平台整体层面，还定义了协同效率指数：

$$
\mathcal{E} = \sum_{i=1}^{n} T_{saved, i} \cdot \omega_{user}
$$

其中 $T_{saved, i}$ 表示第 $i$ 次解析节省时间，$\omega_{user}$ 表示用户权重。该指数用于观察平台在持续使用过程中的累计效率增益，并可结合 ECharts 趋势图进行展示。

除上述公式外，系统还利用 jieba 与停用词过滤对材料词频、风险词频与主题要素进行聚合统计，并按顺序或按小时聚合 RAG 查询次数、命中率、平均相似度和空检率，为个人中心、管理后台和展示大屏提供数据底座。

### 3.10 重点难点与解决方案

#### 3.10.1 自由格式图谱解析难点

政策与通知材料结构差异大，若采用固定字段模板，会在复杂附件、图文混排、步骤嵌套和条件限制较多的场景下丢失关系信息。为解决这一问题，系统不直接追求少量固定字段的“完整命中”，而是采用自由格式图谱解析思路，将结果转为 `dynamic_payload` 并进一步反推图谱。

对应解决方案包括：

- 通过 `json_schema / json_object` 优先约束大模型输出  
- 对不规范 JSON 执行对象提取、冗余逗号修复和截断补全  
- 在轻量恢复失败后进入 Repair-Only 二次修复  
- 在模型只输出少量抽象节点时，基于原文行块和章节结构补出原文结构图层  

通过上述方式，系统能够在材料结构复杂、表达方式不统一的前提下，依然给出最小可展示、可联动、可继续追问的图谱结果。

#### 3.10.2 实时轨迹展示难点

智能体推理步骤产生速度与前端渲染节奏并不一致。若简单地将后端事件按生成顺序直接推送，前端会出现轨迹闪过、正文覆盖轨迹或最后几步来不及展示的问题，严重影响可解释性。

系统的解决方式是：

- 在后端使用 `asyncio.Queue + asyncio.to_thread` 解耦推理和推送  
- 把工具调用、思考事件、正文结果拆分成不同阶段事件  
- 通过 `trace_done` 信号显式划分“轨迹结束”和“正文开始”  
- 在前端对轨迹消费设置最小可读时长，确保步骤能够被稳定看清  

这一方案的关键价值不在于单纯流式输出，而在于让“推理过程可读”成为稳定体验的一部分。

#### 3.10.3 多模态文档处理难点

不同材料来源的解析链路差异极大。PDF 可能包含文本层、图片层或扫描页；DOCX 可能以段落为主，也可能嵌入大量图片；截图和扫描件则更依赖 OCR 质量。若所有材料都采用同一提取策略，系统稳定性会明显下降。

系统为此采用“文本优先、混合提取、回退容错”的策略：

- PDF 优先提取文本块，并按页面阅读顺序混排文本与图片块  
- 图片、截图和扫描件进入 OCR 流程，再统一标准化文本  
- 对多图文档和复杂附件采用分流策略，避免串行 OCR 造成严重阻塞  
- 对提取结果做文本清理和格式规范化，确保后续自由格式图谱解析的输入一致性  

该设计使多模态能力不是分裂的附加功能，而是统一服务于后续理解链路的稳定基础。

#### 3.10.4 可信度保障难点

在复杂通知和规则说明场景下，用户最担心的不是回答“够不够快”，而是回答“有没有依据、能不能核对、是否容易误导”。通用问答工具虽然可以生成看似流畅的解释，但如果没有证据约束，结果难以直接用于执行决策。

系统的应对方式包括：

- 使用本地知识库与 ChromaDB 构建稳定的语义检索基础  
- 通过 `top_k + score_threshold` 双阈策略过滤低质量片段  
- 在前端显式展示证据片段与相似度分数  
- 在后台持续记录命中率、平均得分和空检率等指标  

因此，可信度问题在本系统中不是“回答后人工判断”，而是“在生成、展示和统计三个层面同步控制”。

#### 3.10.5 结果沉淀与后续承接难点

很多同类系统在给出回答后即结束，用户无法方便地继续追问、回看、收藏或转化为下一步事项，从而导致同一份材料在后续使用中被重复阅读、重复整理。对真实使用过程来说，这会削弱系统长期价值。

本作品通过以下方式解决这一问题：

- 将解析结果接入收藏、历史记录和待办事项模块  
- 通过 `HistoryEvent` 统一记录跨页面操作与恢复入口  
- 允许解析结果进一步转化为待办草稿并进行确认管理  
- 通过 `Favorite` 和会话持久化保留个人长期可复用资产  

由此，解析结果不再只是一次展示，而成为后续使用与持续沉淀的起点。

#### 3.10.6 多角色协同与平台化扩展难点

若系统只面向单一用户，产品结构相对简单；但本作品同时服务普通用户、认证主体和管理员，要求在一套系统中兼顾个人使用、公开发布、反馈治理和后台管理。若缺少清晰边界，系统很容易变成页面堆叠。

本作品的解决方式是：

- 在身份层使用 `role` 明确区分三类角色  
- 在业务层将解析、发布、反馈和管理拆分成独立模块  
- 在展示层保留 `/showcase` 与业务入口的连续访问链路  
- 在数据层通过文档、意见、统计和历史表保持角色操作的可追踪性  

这使系统既保持了平台化结构，又没有丢失核心主线。

### 3.11 安全性与权限控制设计

系统采用 JWT 鉴权机制，结合用户角色进行权限控制。普通用户、认证主体和管理员具备不同操作能力：

- 普通用户：解析、追问、收藏、待办、查看公开内容、提交意见
- 认证主体：在普通用户能力基础上增加发布与追踪能力
- 管理员：增加审核、用户管理、数据分析和系统监控能力

前端通过路由守卫限制管理员页面访问，后端通过依赖注入与鉴权逻辑控制实际接口权限，保证角色边界一致。

### 3.12 可部署性与可扩展性设计

系统支持本地开发部署与 Docker 部署两种运行方式。Nginx 负责统一代理前端静态资源、后端 API、媒体文件和 WebSocket 通信，Redis 负责缓存与异步任务。该设计具备以下优势：

- 便于本地快速演示与开发
- 部署结构清晰，适合迁移到服务器环境
- Agent、OCR、新闻抓取和统计分析等模块可以独立扩展
- 展示层与业务层位于同一产品体系内，便于后续继续扩展新页面和新角色能力

### 5.4 公网部署
1. 准备云服务器
  - 系统建议 Ubuntu 22.04/24.04。
  - 安装 Docker 和 Compose 插件 和 Git。
   ```bash
   sudo apt update
   sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin git

   ```
   注意：如果你之前装过 `docker.io`，可能出现 `containerd.io conflicts: containerd`。这种情况下不要混装，统一使用 Docker 官方这一套即可。
2. 阿里云安全组放行 80
   如果服务器本机容器正常，但浏览器打不开 `http://47.86.23.163/`，最常见原因不是代码，而是云安全组没有放行 80。
   本次真实配置为：
   - 授权策略：`允许`
   - 优先级：`1`
   - 协议类型：`TCP`
   - 访问来源：`IPv4`
   - 源地址：`0.0.0.0/0`
   - 访问目的：本实例
   - 端口：`HTTP(80)`
   - 描述：`http`
   如后续要上 HTTPS，再加一条 `HTTPS(443)`。

3. 在服务器中 clone 项目源代码
    ``` bash
    cd /opt
    sudo git clone https://github.com/xiao-Huahuo/CloudPolicy.git cloudpolicy
    sudo chown -R $USER:$USER /opt/cloudpolicy
    cd /opt/cloudpolicy
    ```
4. 在服务器上创建环境变量文件
   在根目录 `/opt/cloudpolicy` 创建生产环境专用的 `.server.env`。
   `.server.env` 建议直接参考上面的环境变量模板填写，字段体系与 `.env` 基本一致；Docker 场景下由 `docker-compose.prod.yml` 自动覆盖 `DOCKER_DEPLOYMENT`、`HOST`、`REDIS_HOST` 等运行时差异。
   若服务器公网 IP 为 `47.86.23.163`，则至少应修改以下字段：
    ```env
    FRONTEND_BASE_URL_DOCKER=http://47.86.23.163
    PUBLIC_BASE_URL_DOCKER=http://47.86.23.163/api
    ```
   另外建议同时确认以下字段：
   - `SECRET_KEY`：替换为新的随机密钥
   - `AUTH_SANDBOX_SMS=false`：公网部署时关闭短信验证码预览
   - `SMTP_*`、`LLM_*`：替换为服务器实际可用的配置
   - `PADDLEOCR_ENABLED`：如首次部署因 OCR 模型准备失败，可先临时设为 `false`

5. 先构建 app 镜像
   ```bash
   docker compose --env-file .server.env -f docker-compose.prod.yml build app
   ```

6. 用 app 镜像把 embedding 下载到运行卷
   这是这次实际部署里最关键的一步。因为后端启动时会检查 embedding 模型，如果直接 `up -d` 而模型还没准备好，`CloudPolicy_Service` 会反复启动失败。
   ```bash
   docker compose --env-file .server.env -f docker-compose.prod.yml run --rm --no-deps app /bin/sh -lc "
     set -eu
     mkdir -p /runtime/resources/embedding
     rm -rf /server/app/resources/embedding
     ln -snf /runtime/resources/embedding /server/app/resources/embedding
     python -m app.scripts.download_embedding
   "
   ```

7. 可选：确认 embedding 已进入运行卷
   ```bash
   docker compose --env-file .server.env -f docker-compose.prod.yml run --rm --no-deps app /bin/sh -lc "
     find /runtime/resources/embedding -maxdepth 2 -type f | head
   "
   ```

8. 正式启动生产容器
   ```bash
   docker compose --env-file .server.env -f docker-compose.prod.yml up -d --build
   docker compose --env-file .server.env -f docker-compose.prod.yml logs -f
   ```
   如果看到：
   - `CloudPolicy_Redis_Service` healthy
   - `CloudPolicy_Service` healthy
   - `CloudPolicy_Web` started
   说明容器层已经启动完成。
   
9. 最终验收
   在浏览器访问：
   - `http://47.86.23.163/`
   - `http://47.86.23.163/api/docs`
   两个地址都能打开，就说明这次公网部署已经真正完成。

10. 关机后重新启动的方法
  1. 确认 Docker 是否起来
  ```bash
  sudo systemctl status docker
  ```
  2. 进入项目目录
  ```bash
  cd /opt/cloudpolicy
  ```
  3. 启动容器
  ```bash
  docker compose --env-file .server.env -f docker-compose.prod.yml up -d
  ```
  4. 看状态
  ```bash
  docker compose --env-file .server.env -f docker-compose.prod.yml ps
  ```
  5. 如果有问题，再看日志
  ```bash
  docker compose --env-file .server.env -f docker-compose.prod.yml logs -f
  ```

## 第六章 项目总结

### 6.1 项目成果总结

云枢观策已经形成一套结构较完整的智能化服务平台，具备展示入口、业务系统入口、多模态解析工作台、可解释智能体、知识图谱展示、政策广场、结果沉淀、反馈大厅、认证发布和管理员后台等模块。作品不再停留在单个算法验证或单页面演示，而是逐步形成面向真实使用过程的产品结构。

### 6.2 作品的应用价值

本作品的价值主要体现在以下几个方面：

- 降低复杂通知、政策说明和附件材料的阅读理解成本
- 提高用户从材料中提取关键步骤、材料清单和时间节点的效率
- 通过可解释智能体增强用户对结果的理解与核对能力
- 将单次解析结果延伸为收藏、待办、历史和反馈等后续操作
- 为认证主体和管理端提供更清晰的发布、追踪和分析能力

作品关注的是信息理解、执行承接和反馈沉淀的完整链路，因此更适合在校园、社区、园区、服务窗口和内容发布场景中扩展使用。

### 6.3 开发过程中的重点突破

在开发过程中，作品的主要难点并不在于单独调用某一项技术，而在于如何把多种能力稳定衔接为一套系统。较为关键的突破包括：

- 形成适应复杂材料的自由格式图谱解析链路
- 建立结果与图谱之间的自动映射关系
- 构建实时推理轨迹流式编排机制
- 将 OCR、RAG、Agent 和统计分析纳入统一产品结构中
- 兼顾展示效果与实际使用逻辑

### 6.4 团队协作与能力提升

项目开发过程中，团队围绕产品设计、前后端开发、智能体接入、RAG 机制、多模态解析、部署适配和文档整理等任务展开分工协作。通过本次作品开发，团队在以下方面获得了较明显提升：

- 更完整地理解了从需求、架构、实现到部署的系统化开发过程
- 在前后端协作、接口设计和模块划分方面形成了更稳定的工程习惯
- 在大模型应用、RAG、OCR 和流式交互方面积累了实践经验
- 在产品展示、结构表达和文档整理方面形成了更成熟的方法

### 6.5 后续优化与扩展方向

后续可围绕以下方向继续完善作品：
